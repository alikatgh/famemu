* Math-chip trig and inverse tables are computed from their definitions, not dumped from silicon. Five core commands match the reference bit-for-bit; two differ by ≤2 low bits and are pinned by golden screens instead. This is the clean-room tradeoff, stated plainly.

The one gap

The SPC7110-class decompressor is not implemented — its ROM banking and real-time clock work, but the context-modeled decompression engine is stubbed and logs on first use. It affects three Japan-only commercial titles; it is irrelevant to writing new games for Ember 16. We list it here rather than quietly omit it, because an honest state page is worth more than a complete-looking one.

How verification works

Three layers, all enforced by ctest on every build. The gates are the documentation's warranty.

Community test cartridges

The standard 6502 / 65816 / audio correctness suites, run to golden CRCs. These are the same cartridges homebrew developers already trust.

Lockstep diffs

Our own feature cartridges rendered by both Ember and a reference implementation of the hardware, compared frame-by-frame. 43 frames plus three full gameplay scenarios, all at 0.000% divergence today.

Golden screens

66 CRC-pinned frames catch any drift — including effects the oracle can't gate (mid-scanline splits, interlace fields, computed-table chip ops). Any change to a verified screen fails the build.

Reference emulators are used only as test oracles. None of their code is linked, derived, or shipped. See the heritage & clean-room policy.
