Two consoles, one contract

Both engines expose the same tiny C interface, so the app, the store, and the tools never care which one is running. Pick the era that fits your game.

Ember 8  — the 8-bit era

An 8-bit 6502-family CPU, a 256×240 tile-and-sprite picture with a 64-color palette, and five audio channels (two pulse, triangle, noise, and sampled DPCM). Cartridge boards from fixed 32K up to the workhorse bank-switcher with a scanline IRQ counter. Sprite-0 splits, mid-frame scroll, raster HUD bars — all exactly as the hardware did them.

Ember 16  — the 16-bit era

A 16-bit 65816 with true master-cycle timing, four background layers across eight video modes (rotation and scaling, genuine 512-wide hi-res, per-scanline and mid-scanline raster effects), 128 sprites, and a real audio subsystem: a dedicated sound CPU driving an 8-voice sample DSP with echo. Cartridges can add math, 3-D, decompression, and dual-CPU chips.

How it works

The engine is the funnel; the store is the business. The code is free so anyone can build on it — the catalog and curation are ours.

1 · Write a cartridge

Assemble with the mature cc65 toolchain against a memory map. Start from the maintained example games and the feature demos — each one boots from reset and exercises one subsystem in ~200 lines.

2 · Run it anywhere

The famemu app runs your image on iOS and macOS. Every engine is deterministic: same cartridge, same input, identical frames on every device. That guarantee is what the whole test pipeline is built on.

3 · Ship on the store

Submit an original game; it's curated, signed, and sold. Engines stay free; games are the paid catalog. The app is a games platform — it runs store-signed content and your own dev builds, nothing else.

Authenticity you can check

"Hardware-authentic" is a testable claim here, not a vibe. Every capability the engines advertise is pinned by an automated gate.

Verified against silicon

The engines render the same test cartridges as a reference implementation of each hardware class, and CI diffs the output frame-by-frame. Current standing: 43 lockstep frames and three full gameplay scenarios at 0.000% divergence, plus 66 CRC-pinned golden screens. If a doc claims it, a gate proves it.

Clean-room and honest about gaps

Every line is original, from public documentation — no dumped chip ROMs, no copyleft emulator code. Reference emulators are used only as test oracles. And we publish the one feature still unfinished (the SPC7110-class decompressor) rather than paper over it. See the state page.
